# 模組 3 作業：IDE 整合 - 工具選擇實戰

## 📋 作業概述

本作業旨在檢驗你對 IDE 整合工具的理解和實際應用能力，重點在於：
- 工具選擇決策能力
- 混合工具鏈設計
- 工作流程優化

**預計時間**：2-3 小時
**截止日期**：課程第 3 週結束
**提交方式**：GitHub Repository 或 Markdown 文件

---

## 🎯 作業目標

完成本作業後，你將能夠：
- ✅ 為不同場景快速選擇最適合的 AI 工具
- ✅ 設計高效的混合工具鏈
- ✅ 優化個人和團隊的開發工作流程
- ✅ 評估工具選擇的成本效益

---

## 📝 作業內容

### 任務 1：工具選擇決策實戰（30 分）

#### 背景

你是一家中型軟體公司的技術顧問，需要為不同的開發場景推薦合適的 AI 工具組合。

#### 場景清單

**場景 1：緊急 Bug 修復**
- **情況**：生產環境出現嚴重 bug，需要在 1 小時內定位並修復
- **專案規模**：50,000 行 Python/Django 專案
- **團隊**：你一個人
- **要求**：快速定位問題、提供修復方案、驗證修復

**場景 2：新功能開發**
- **情況**：開發一個新的使用者認證模組
- **專案規模**：新模組，預計 2,000 行程式碼
- **團隊**：2 人協作
- **要求**：快速開發、安全可靠、完整測試

**場景 3：代碼庫現代化**
- **情況**：將舊版 React Class Components 升級為 Function Components + Hooks
- **專案規模**：200 個元件檔案，約 80,000 行程式碼
- **團隊**：5 人團隊
- **要求**：保持一致性、避免遺漏、確保功能不變

**場景 4：技術債務清理**
- **情況**：重構一個複雜的資料處理模組，消除重複程式碼
- **專案規模**：15,000 行程式碼，涉及 30 個檔案
- **團隊**：你一個人
- **要求**：改善程式碼品質、不破壞現有功能、提升可維護性

**場景 5：開源專案貢獻**
- **情況**：首次貢獻一個大型開源專案（Kubernetes）
- **專案規模**：500,000+ 行 Go 程式碼
- **團隊**：獨立貢獻者
- **要求**：快速理解架構、遵守專案規範、提交高品質 PR

**場景 6：API 文檔生成**
- **情況**：為現有的 REST API 生成 OpenAPI 文檔
- **專案規模**：120 個 API 端點，Flask 專案
- **團隊**：你一個人
- **要求**：準確完整、符合 OpenAPI 規範、包含範例

**場景 7：性能優化**
- **情況**：分析並優化資料庫查詢效能
- **專案規模**：300 個 SQL 查詢，分散在 50 個檔案中
- **團隊**：你和 DBA 各一人
- **要求**：找出慢查詢、提出優化方案、驗證改進效果

**場景 8：測試覆蓋率提升**
- **情況**：為現有專案添加單元測試，目標覆蓋率 80%
- **專案規模**：20,000 行業務邏輯程式碼，當前覆蓋率 30%
- **團隊**：3 人團隊
- **要求**：快速編寫測試、覆蓋關鍵路徑、高品質測試

**場景 9：微服務拆分**
- **情況**：將單體應用拆分為微服務架構
- **專案規模**：150,000 行程式碼的 Monolith
- **團隊**：8 人團隊
- **要求**：識別服務邊界、規劃遷移策略、最小化風險

**場景 10：知識文檔撰寫**
- **情況**：為團隊撰寫系統架構文檔和開發指南
- **專案規模**：複雜的微服務系統
- **團隊**：你一個人
- **要求**：清晰易懂、包含圖表、實用性強

### 任務 2：混合工具鏈設計（40 分）

#### 背景

你的團隊（15 人研發團隊）希望全面導入 AI 工具，需要設計一套完整的工具鏈和使用規範。

#### 團隊資訊

**團隊結構**：
- 前端團隊：6 人（React + TypeScript）
- 後端團隊：6 人（Python + FastAPI）
- DevOps：2 人
- 技術主管：1 人

**技術棧**：
- 前端：React 18, TypeScript, Vite, TailwindCSS
- 後端：Python 3.11, FastAPI, PostgreSQL, Redis
- 基礎設施：Docker, Kubernetes, GitHub Actions
- 監控：Prometheus, Grafana, ELK Stack

**目前痛點**：
1. 程式碼品質不一致（每人風格不同）
2. 重複工作多（相似功能重複實作）
3. 新人 Onboarding 慢（需要 2-3 週理解系統）
4. 文檔更新滯後（與程式碼不同步）
5. 測試覆蓋率低（約 40%）

**預算**：每月 $500（約 $33/人）

**目標**：
- 提升開發效率 50%
- 改善程式碼品質（減少 bug 30%）
- 縮短新人 Onboarding 時間到 1 週
- 提升測試覆蓋率到 70%

### 常見問題

**Q1：可以使用 AI 工具完成作業嗎？**

A：可以，而且**強烈建議使用**！本作業的目的就是實踐 AI 工具的使用。

**建議做法**：
1. 使用 AI 協助分析場景
2. 讓 AI 提供工具選擇建議
3. 你來評估和最終決策
4. 在提交中說明哪些部分使用了 AI 輔助

---

**Q2：團隊工具鏈設計沒有實際團隊怎麼辦？**

A：可以：
- 基於虛擬團隊設計（使用作業提供的團隊資訊）
- 基於你過去的團隊經驗
- 基於你理想中的團隊配置

重點是**設計思路**和**決策邏輯**，而非必須有實際團隊。

---

**Q3：預算不夠怎麼辦？**

A：這是很現實的挑戰！建議：
- 優先配置核心開發人員
- 考慮免費或低成本替代方案
- 設計分階段導入計劃
- 說明如何在有限預算內最大化效益

**成本控制本身就是評分點之一**。

---

**Q4：個人工作流程優化要多詳細？**

A：建議包含：
- 至少 3 個主要工作場景
- 每個場景的 Before/After 對比
- 具體的時間估算
- 可驗證的評估指標

不需要完美，但要**真實且可執行**。
