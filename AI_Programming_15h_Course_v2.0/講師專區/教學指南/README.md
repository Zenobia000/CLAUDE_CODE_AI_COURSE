## 🎯 教學目標

### 核心理念

本課程採用**情境驅動學習法 (Scenario-Driven Learning)**，而非傳統的指令式教學：

```
傳統教學 ❌:
"今天學習 /search 指令，它的語法是..."

情境驅動教學 ✅:
"你的專案出現 bug，你懷疑是某個函數導致的，
但專案有 50 個檔案。你會怎麼快速找到它？"
→ 學生自然學到 /search 指令
```

### 教學哲學（基於 Linus Torvalds 的學習理念）

1. **不教指令，教方向**
   - 如同學 Linux 不是背 ls/cd/grep，而是知道「遇到問題時往哪個方向探索」
   - 學生應該學會「問對問題」，而非「記住答案」

2. **從問題到解決方案**
   - 每個概念都從真實問題出發
   - 指令是解決問題的副產品，不是學習目標

3. **驗證思維取代生成思維**
   - 不是「讓 AI 幫我寫代碼」
   - 而是「讓 AI 幫我檢查代碼是否安全、高效、可維護」

## 🎓 教學方法論

### 1️⃣ 情境引入法

**步驟**:

1. **設定情境**: 描述真實開發場景
   ```
   "你接手一個 3 年前的專案，沒有文檔，
   需要找出所有 API endpoint 的定義位置..."
   ```

2. **提出挑戰**: 明確的問題
   ```
   "如何在 30 秒內找到所有路由定義？"
   ```

3. **引導探索**: 不直接給答案
   ```
   "如果是你，你會怎麼做？"
   → 討論學生想法
   → 引導到工具使用
   ```

4. **演示解決方案**: 展示高效方法
   ```
   "讓我示範如何用 Claude Code 快速解決..."
   ```

5. **反思總結**: 提煉學習點
   ```
   "這個情境中，你學到的不只是 /search 指令，
   更重要的是『大型專案中定位問題的思維』"
   ```

### 2️⃣ 對比教學法

**傳統 vs AI 輔助對比**:

```python
# 傳統方式（10 分鐘）
# 1. 手動打開每個檔案
# 2. Ctrl+F 搜尋關鍵字
# 3. 記錄找到的位置
# 4. 重複 1-3 步驟

# AI 輔助方式（30 秒）
# 1. /search "router.get"
# 2. 查看結果列表
# 3. 完成！
```

**關鍵**: 強調時間節省與認知負擔減輕

### 3️⃣ 錯誤驅動教學法

**刻意展示錯誤用法**:

```python
# ❌ 學生常見錯誤
"幫我寫一個用戶認證系統"
→ AI 生成大量代碼
→ 學生不理解，直接複製貼上
→ 出現 bug 時不知如何修

# ✅ 正確方法
"我寫了一個 JWT token 驗證函數（貼上代碼），
請檢查是否有安全漏洞"
→ AI 指出 secret key 硬編碼問題
→ 學生理解並修正
→ 學到安全最佳實踐
```

## 📊 評量與反饋

### 形成性評量（過程中）

**情境完成度檢查清單**:

```markdown
學員完成情境後，檢查：
- [ ] 是否理解問題本質？
- [ ] 使用的工具是否恰當？
- [ ] 能否解釋解決方案？
- [ ] 遇到錯誤時如何處理？
```

**即時反饋技巧**:

```
❌ 不好的反饋: "這樣不對，應該用 /edit"
✅ 好的反饋: "你用 Write 工具很好，但如果檔案已存在，
              用 Edit 會更安全。為什麼？試試看會發生什麼。"
```

### 總結性評量（課程結束）

參考 `11_實戰綜合項目/評估標準/` 中的評分表。

**評估維度**:

1. **技術能力** (40%)
   - AI 工具使用熟練度
   - 代碼品質
   - 問題解決能力

2. **思維方式** (30%)
   - 情境分析能力
   - 提示詞設計能力
   - 驗證優先思維

3. **實作成果** (30%)
   - 專案完成度
   - 文檔品質
   - CI/CD 設置

## 💡 常見教學挑戰與解決方案

### 挑戰 1: 學員過度依賴 AI 生成

**症狀**:
```
學員直接問 AI "幫我寫整個登入系統"
然後複製貼上所有代碼
```

**解決方案**:
```
1. 設定規則: "先寫測試，再用 AI 輔助實作"
2. 要求學員解釋每段 AI 生成的代碼
3. 演示驗證導向的工作流程:
   - 自己寫初版
   - AI 審查並建議
   - 理解建議後修改
   - 再次請 AI 驗證
```

### 挑戰 2: 學員不會問問題

**症狀**:
```
學員提示詞: "這段代碼有問題嗎？"
→ AI 回覆過於籠統
```

**解決方案**:

**教導 STAR 提示詞框架**:

```markdown
S (Situation): 背景情境
"我在開發一個任務管理 API，使用 FastAPI + SQLAlchemy"

T (Task): 具體任務
"需要實作用戶認證中間件，驗證 JWT token"

A (Action): 已完成的動作
"我寫了以下代碼（貼上代碼）"

R (Result): 期望結果
"請檢查是否有安全漏洞，特別是 token 過期處理"
```

### 挑戰 3: 進度差異大

**症狀**:
```
有些學員 30 分鐘完成情境
有些學員 2 小時還在掙扎
```

**解決方案**:

**採用分層教學**:

```
快速完成者:
→ 提供進階挑戰（複雜級情境）
→ 邀請協助其他學員（教學相長）

需要更多時間者:
→ 提供更詳細的步驟提示
→ 安排 1-on-1 指導時間
→ 簡化情境複雜度
```

## 🎯 課程進度監控

### 每週檢查點

| 週次 | 模組 | 關鍵檢查點 |
|------|------|-----------|
| Week 1-2 | Module 0-1 | 學員能獨立使用 Claude Code 基本功能 |
| Week 3-4 | Module 2-2.5 | 熟練使用情境指令與上下文管理 |
| Week 5-6 | Module 3-4 | 能在 IDE 中高效協作開發 |

| Week 9-10 | Module 7-8 | 掌握 CI/CD 與 DDD 概念 |
| Week 11-12 | Module 9-10 | 完成性能優化與安全掃描 |
| Week 13-14 | Module 11 | 完成綜合專案 |

### 進度落後補救措施

```markdown
如果班級整體進度落後：
1. 減少理論講解，增加實作時間
2. 將複雜級情境改為課後作業
3. 延長關鍵模組的教學時間

如果個別學員落後：
1. 提供課後輔導時間
2. 錄製重點內容供複習
3. 簡化作業要求
```
